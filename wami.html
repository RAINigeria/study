SYSTEM
INTRODUCTION

Wide-Area Motion Imagery (WAMI) is a surveillance approach that uses an airborne array of cameras to image an entire city-scale area simultaneously, at video frame rates, continuously — enabling operators to detect and track every moving vehicle and person within the coverage zone in real time, and to rewind and forensically trace any movement backwards in time.

Unlike a conventional camera that can only stare at one narrow patch of ground at a time, a WAMI payload stares at everything at once. This is the key capability that makes it operationally revolutionary — no subject can be lost simply by moving out of frame.

// Core Principle

WAMI does not trade resolution for coverage in the way a single wide-angle camera would. It achieves both by using many sensors in parallel, each covering a tile of the total area at acceptable resolution. The mosaic of tiles gives you the city. Each tile gives you the detail.

WHAT THIS TUTORIAL COVERS

This tutorial walks through every engineering layer of a WAMI system from scratch — the optics, the sensor electronics, the mechanical arrangement, the processing pipeline, and the calibration procedure. By the end you will have enough to specify, assemble, and operate a functional prototype.

// 02.02

CAMERA
ARRAY DESIGN

The defining feature of a WAMI payload is that it uses many cameras in a mosaic arrangement, not a single large camera. This is driven by both physics and economics — there is no single sensor with enough pixels to achieve both city-scale coverage and 30cm ground resolution simultaneously. The mosaic approach also allows cameras to point obliquely outward, dramatically increasing coverage area per unit altitude.

RING ARRANGEMENT

For 360° coverage (aerostat-based systems like Kestrel), cameras are arranged in an outward-facing ring, each pointed radially outward at a steep oblique angle. For directional systems (fixed-wing aircraft like PSS Cessna), cameras are arranged in a downward/forward-facing mosaic through the aircraft belly.

// 8-Camera Ring Arrangement — Top View

SIDE VIEW — OBLIQUE POINTING ANGLE

// Single Camera — Oblique Viewing Geometry

OVERLAP ZONES

Adjacent cameras must overlap by at least 15–20% of their field of view. This overlap serves two purposes: it eliminates gaps in coverage, and it provides the redundant features needed by the online calibration algorithm to detect and correct any inter-camera misalignment.

// Design Rule

More overlap = better calibration stability, but reduced net coverage area per camera. 15–20% is the standard engineering compromise. Below 10% and calibration becomes unreliable; above 30% and you're wasting pixel budget on redundant coverage.

FOCAL PLANE
ARRAYS

A focal plane array (FPA) is the core sensor chip inside each camera. It is a 2D grid of photodetectors — each detector is one pixel. When light falls on a detector, it generates an electric charge proportional to the number of photons received. That charge is read out as a voltage and digitised to give a pixel value.

// Sensor Selection for Prototype

For a first prototype, start with Sony IMX sensors (IMX294, IMX410, or IMX455) — large format, high resolution, well-documented, available without NDA. The IMX455 gives 61MP at 3.76µm pixel pitch in a 35mm format, excellent for high-altitude work. For IR capability later, an uncooled LWIR microbolometer like the FLIR Lepton or Seek Mosaic can be added to alternate camera slots.

BAYER PATTERN & DEMOSAICING

Most colour CMOS sensors use a Bayer colour filter array — each pixel is covered by either a red, green, or blue filter in a 2×2 RGGB pattern. The raw output is therefore a Bayer image, not a full colour image. Before use, the image must go through demosaicing — an interpolation algorithm that reconstructs full RGB colour at every pixel from its neighbours. This is part of the processing pipeline and must be accounted for in compute budget.

// 02.04

TELESCOPIC
OPTICS

The lens is the component that determines ground resolution. Given a sensor with known pixel pitch and a required ground sample distance (GSD) at a given slant range, the required focal length can be calculated exactly.

FOCAL LENGTH CALCULATION

// Fundamental Lens Equation for Aerial Work
focal_length = (pixel_pitch × slant_range) / GSD
// Example: Sony IMX455 at 1km altitude, 70° off-nadir
pixel_pitch = 3.76 µm (0.00376 mm) slant_range = 1000m / cos(70°) = 2924m target GSD = 0.35m (35cm) focal_length = (0.00376 mm × 2,924,000 mm) / 350 mm = 31.4mm
// Note

A 31mm focal length for a 35mm-format sensor is a moderate telephoto — similar to a 50mm equivalent on a full-frame camera. At 2km altitude the same calculation gives ~63mm. At 3km altitude ~94mm. These are all achievable with standard M42 or C-mount machine vision lenses, not exotic military optics.

LENS QUALITY REQUIREMENTS

For WAMI use, the lens must be:

  • Sharp corner-to-corner — vignetting and corner softness that are acceptable in photography degrade mosaic stitching quality badly
  • Low distortion — barrel/pincushion distortion must be characterised and corrected; the geo-registration algorithm needs a rectilinear projection
  • Fixed focal length — no zoom, no autofocus mechanism. Set to infinity on the ground, lock with thread adhesive (Loctite)
  • Thermally stable — focus shift with temperature must be <1 depth-of-field unit across operating temperature range (−10°C to +40°C typically)
  • Matched set — all 8 lenses should be the same model and ideally same production batch to minimise inter-camera colour and vignetting differences

DEPTH OF FIELD AT ALTITUDE

A key advantage of aerial photography: at 1km+ altitude, depth of field is effectively infinite for any practical focal length. Focus set once at infinity covers everything from horizon to horizon. This eliminates autofocus as a concern entirely.

// Hyperfocal distance — beyond this, everything is sharp to infinity
H = f² / (N × c) where: f = focal length, N = f-number, c = circle of confusion (~2 × pixel pitch) Example (31mm lens, f/4, IMX455): c = 2 × 0.00376mm = 0.00752mm H = (31)² / (4 × 0.00752) = 961 / 0.030 = ~32,000mm ≈ 32m At 1km altitude everything beyond 32m from the lens is in focus. // Conclusion: focus once, lock, forget.

GIMBAL &
STABILISATION

The gimbal is the mechanical decoupling layer between the aircraft (which vibrates, pitches, rolls, and yaws) and the sensor payload (which needs to point at a fixed patch of ground). Without stabilisation, imagery is blurred and geo-registration fails.

STABILISATION REQUIREMENTS

Disturbance SourceFrequencyMitigation
Engine/motor vibration50–200 HzVibration isolators (rubber dampers) between payload and airframe
Wind gusts0.1–2 Hz3-axis gyrostabilised gimbal with rate feedback
Platform attitude changes0.01–0.5 HzIMU-referenced gimbal, GPS/INS integration
Thermal expansionDC / very slowInvar or carbon fibre structural elements, online calibration

GIMBAL ARCHITECTURE

A 3-axis brushless motor gimbal provides roll, pitch, and yaw correction. IMU sensors on the payload continuously measure attitude error; a PID controller drives the gimbal motors to null the error. Modern drone-grade gimbals (e.g. Gremsy H16, DJI RS Pro) achieve ±0.01° stabilisation, more than adequate for WAMI work.

// Critical Note

For a ring camera array, the gimbal must stabilise the entire ring assembly as a rigid body. Do not attempt to stabilise individual cameras. The inter-camera geometry must be preserved rigidly — the gimbal corrects the whole payload's attitude, not each camera independently.

// Numerical example: h = 1000m, θ = 70°

// 03.02

COS θ
CORRECTION

When a camera points obliquely at the ground, each pixel covers a different-sized area depending on direction. In the range direction (along the viewing direction projected to ground), pixels are stretched by a factor of 1/cos(θ). In the azimuth direction (perpendicular), they are much less affected. This creates a non-square pixel footprint that must be corrected before mosaicking.

GSD IN EACH DIRECTION

// Azimuth GSD (perpendicular to look direction — less distorted)
GSD_az = (p × R) / f = (p × h) / (f × cos θ)
// Range GSD (along look direction — stretched by oblique angle)
GSD_r = GSD_az / cos(θ) = (p × h) / (f × cos²θ)
// Numerical example: p=3.76µm, f=31mm, h=1000m, θ=70°
GSD_az = (0.00376mm × 1,000,000mm) / (31mm × cos(70°)) = 3760 / (31 × 0.342) = 3760 / 10.6 = 355mm ≈ 35.5cm GSD_r = 355mm / cos(70°) = 355 / 0.342 = 1038mm ≈ 104cm // A pixel that is 35cm wide is nearly 1m deep in the range direction. // This is why resolution degrades with oblique angle — range pixels are stretched.

ORTHORECTIFICATION

To correct for this stretch before mosaicking, each image tile must be orthorectified — projected onto a flat horizontal ground plane as if viewed from directly above. This is done using the known camera position (from GPS), attitude (from IMU), and a digital elevation model (DEM) of the terrain.

01

Build Ray from Camera to Ground

For each pixel in the raw image, compute the 3D ray from the camera centre through that pixel using the known focal length and pixel position. Transform to world coordinates using the GPS position and IMU attitude.

02

Intersect Ray with Ground Plane

Find where each ray intersects the terrain. If a DEM is available, use it; otherwise assume flat ground at the average altitude of the scene. The intersection gives the ground coordinate (latitude, longitude) of that pixel.

03

Resample to Regular Grid

Map pixel values from the raw (stretched) image to a regular UTM or WGS84 grid at the target GSD. Use bilinear or bicubic interpolation. The output is a georeferenced image tile with square pixels at the nominal GSD.

04

Apply Lens Distortion Correction

Before or after ray-casting, correct for radial and tangential lens distortion using pre-calibrated coefficients (k1, k2, p1, p2). Use OpenCV's undistort() or equivalent. This is critical for accurate mosaicking.

import numpy as np

def ray_to_ground(px, py, f, cam_pos, cam_R, h_ground):
    """
    px, py   : pixel coordinates (centred)
    f        : focal length in pixels (f_mm / pixel_pitch_mm)
    cam_pos  : [X, Y, Z] camera position in world coords (m)
    cam_R    : 3x3 rotation matrix from IMU (body to world)
    h_ground : ground height (m, from DEM or assumed flat)
    returns  : (lat, lon) or (X, Y) ground coordinate
    """
    # Ray in camera frame (normalised)
    ray_cam = np.array([px, py, f])
    ray_cam = ray_cam / np.linalg.norm(ray_cam)

    # Transform to world frame using IMU rotation
    ray_world = cam_R @ ray_cam

    # Scale to intersect ground plane at h_ground
    # Parametric: P = cam_pos + t * ray_world
    # At ground: P[2] = h_ground
    t = (h_ground - cam_pos[2]) / ray_world[2]

    ground_point = cam_pos + t * ray_world
    return ground_point[0], ground_point[1]  # X, Y ground

GSD
CALCULATIONS

Ground Sample Distance is the single most important design parameter — it determines what you can and cannot see. This section gives you the complete toolkit to calculate GSD for any combination of sensor, lens, altitude, and pointing angle.

COMPLETE GSD FORMULA SET

// Nadir GSD (straight down, θ = 0)
GSD = (pixel_pitch × altitude) / focal_length
// Oblique GSD — Azimuth direction
GSD_az = (pixel_pitch × altitude) / (focal_length × cos θ)
// Oblique GSD — Range direction (always larger)
GSD_r = (pixel_pitch × altitude) / (focal_length × cos² θ)
// Effective GSD (geometric mean — used for spec purposes)
GSD_eff = sqrt(GSD_az × GSD_r) = (pixel_pitch × altitude) / (focal_length × cos^1.5 θ)

// Takeaway

To maintain acceptable resolution at larger oblique angles you need either a longer focal length lens or you accept lower resolution in the range direction. WAMI systems deliberately operate where the range GSD is 2–3× the azimuth GSD — tracking moving objects (which have extent in both dimensions) tolerates this asymmetry well.

COVERAGE vs
RESOLUTION

Coverage per camera is determined by the total sensor pixel count and the per-pixel GSD.

// Ground coverage of one camera (azimuth direction, in pixels)
Width_ground = sensor_width_pixels × GSD_az Height_ground = sensor_height_pixels × GSD_az (simplified — ignoring range stretch) Area ≈ Width_ground × Height_ground

THE FUNDAMENTAL TRADEOFF SURFACE

// Coverage vs Resolution vs Altitude — Design Space

WAMI systems operate in the 60–75° off-nadir zone to maximise coverage. The penalty is significant range GSD degradation, but this is acceptable for the tracking mission because moving objects are large enough (vehicles are >2m) to be detected even at 1m GSD.

MIPI CSI-2
INTERFACE

MIPI CSI-2 (Camera Serial Interface 2) is the standard digital interface between a CMOS image sensor and a host processor. Understanding this interface is essential for connecting your bare sensors to a compute board.

CSI-2 PHYSICAL LAYER

CSI-2 uses differential serial lanes — each lane is a pair of wires (+ and −) that carry a high-speed differential signal. This gives noise immunity and allows very high data rates. A typical camera uses 2 or 4 data lanes plus one clock lane.

// MIPI CSI-2 Pin Connections

DATA RATE CALCULATION

// Data rate per camera (raw, uncompressed)
data_rate = width × height × fps × bit_depth Example (IMX455, 9576 × 6388, 10fps, 12-bit): = 9576 × 6388 × 10 × 12 = 7.35 Gbits/s per camera 8 cameras total: = 8 × 7.35 = 58.8 Gbits/s // CSI-2 with 4 lanes at 2.5Gbps/lane = 10Gbps per camera port // Need 8 separate CSI-2 ports or FPGA-based aggregation
// Multi-Camera Challenge

This is why Raspberry Pi is only a prototyping tool for one camera. For 8 simultaneous cameras you need either a Jetson AGX Orin (up to 16 CSI lanes, handles ~4 high-res cameras) or an FPGA (Xilinx Zynq, Intel Agilex) that aggregates all 8 streams before feeding a CPU for processing. Real WAMI systems use custom FPGAs precisely for this aggregation layer.

GEO-
REGISTRATION

Geo-registration is the process of assigning a geographic coordinate (latitude, longitude, altitude) to every pixel in every camera frame. It transforms raw imagery from camera space to world space, which is what makes mosaicking and forensic tracking possible.

SENSOR FUSION: GPS + IMU

Accurate geo-registration requires knowing, at the moment of each frame capture:

  • Position — from RTK GPS, ideally to <10cm accuracy
  • Attitude — roll, pitch, yaw from IMU/INS, to <0.05°
  • Camera boresight — the pre-calibrated offset between IMU axes and camera optical axis
  • Timestamp — GPS-synchronised, to <1ms accuracy
python
# Geo-registration: assign lat/lon to each pixel
import numpy as np
from scipy.spatial.transform import Rotation

def pixel_to_latlon(px, py, cam_params, platform_state, terrain_h=0):
    """
    px, py         : pixel coords (0-centred)
    cam_params     : dict with focal_length, pixel_pitch, boresight_R
    platform_state : dict with lat, lon, alt, roll, pitch, yaw (radians)
    terrain_h      : ground elevation (m ASL)
    """
    f_pixels = cam_params['focal_length_mm'] / cam_params['pixel_pitch_mm']

    # 1. Build unit ray in camera frame
    ray_cam = np.array([px, py, f_pixels], dtype=float)
    ray_cam /= np.linalg.norm(ray_cam)

    # 2. Camera to body frame (boresight rotation)
    ray_body = cam_params['boresight_R'] @ ray_cam

    # 3. Body to NED (navigation) frame using IMU angles
    R_body2ned = Rotation.from_euler(
        'ZYX',
        [platform_state['yaw'],
         platform_state['pitch'],
         platform_state['roll']]
    ).as_matrix()
    ray_ned = R_body2ned @ ray_body

    # 4. Scale ray to intersect terrain height
    alt_agl = platform_state['alt'] - terrain_h
    if ray_ned[2] >= 0:
        return None  # Ray points up, no intersection
    t = alt_agl / (-ray_ned[2])
    offset_north = ray_ned[0] * t  # metres north
    offset_east  = ray_ned[1] * t  # metres east

    # 5. Convert offsets to lat/lon
    lat = platform_state['lat'] + (offset_north / 111320)
    lon = platform_state['lon'] + (offset_east / (
          111320 * np.cos(np.radians(platform_state['lat']))))

    return lat, lon

MOSAICKING

Mosaicking assembles the 8 individual geo-registered camera tiles into one seamless wide-area image. After orthorectification, each tile is referenced to the same coordinate grid — mosaicking is therefore primarily about handling the overlap zones between adjacent cameras.

SEAM HANDLING STRATEGIES

Multi-band blendingBlend low frequencies broadly, high frequencies sharplyGood — smooth with preserved detailMedium
Seam-finding + blendingFind optimal seam path, blend either sideBest — invisible seamsHigh
python
# Simple alpha blend across overlap zone
# tiles: list of (image, geo_transform) tuples
import numpy as np

def blend_tiles(tiles, output_bounds, output_gsd):
    """Merge list of (image, geo_transform) tiles into one mosaic"""
    w = int((output_bounds[2] - output_bounds[0]) / output_gsd)
    h = int((output_bounds[3] - output_bounds[1]) / output_gsd)

    mosaic     = np.zeros((h, w, 3), dtype=np.float32)
    weight_map = np.zeros((h, w),    dtype=np.float32)

    for tile_img, geo_tf in tiles:
        # Place tile in mosaic using geo transform
        row0, col0 = world_to_pixel(geo_tf, output_bounds, output_gsd)

        th, tw = tile_img.shape[:2]
        # Build distance-to-edge weight (feathering)
        dist_x = np.minimum(np.arange(tw), tw - np.arange(tw)).astype(float)
        dist_y = np.minimum(np.arange(th), th - np.arange(th)).astype(float)
        weight = np.outer(dist_y, dist_x)
        weight = weight / (weight.max() + 1e-6)

        r1, r2 = row0, row0 + th
        c1, c2 = col0, col0 + tw

        mosaic    [r1:r2, c1:c2] += tile_img * weight[..., np.newaxis]
        weight_map[r1:r2, c1:c2] += weight

    # Normalise by accumulated weights
    mask = weight_map > 0
    mosaic[mask] /= weight_map[mask, np.newaxis]
    return mosaic.astype(np.uint8)
// 05.04

OBLIQUE
CORRECTION

Beyond the cos θ pixel stretch, oblique imagery introduces additional effects that must be corrected for high-quality mosaicking:

1. RELIEF DISPLACEMENT

Tall objects (buildings, trees) appear to lean away from nadir. A building 20m tall at 3km range and 70° off-nadir will appear displaced by 20m × tan(70°) = 55m from its true base position. Correction requires a digital surface model (DSM) with building heights, not just terrain elevation.

2. BIDIRECTIONAL REFLECTANCE (BRDF)

Ground surfaces reflect light differently depending on viewing angle and sun position. Oblique cameras looking sunward see brighter imagery than those looking away. Correct using radiometric calibration: capture grey calibration targets on the ground before flight, or apply relative normalization across the mosaic.

3. ATMOSPHERIC PATH LENGTH

At 70° off-nadir, the atmospheric path is 1/cos(70°) = 2.9× longer than nadir. This increases haze and reduces contrast for the oblique cameras. Apply atmospheric correction or simply apply a contrast/haze correction per camera as a function of off-nadir angle.

// Simplified atmospheric correction
corrected_pixel = raw_pixel × exp(k × (1/cos(θ) - 1)) where k = atmospheric coefficient (~0.05-0.15 depending on visibility)
// 05.05

ONLINE
CALIBRATION

Even with rigid mechanical mounting, inter-camera alignment drifts during flight due to vibration and thermal effects. Online calibration continuously monitors and corrects this drift without interrupting operation.

OVERLAP ZONE FEATURE MATCHING

01

Extract features in overlap zones

Every N frames (e.g. N=10), extract keypoints from the overlap region between adjacent cameras using SIFT, ORB, or SuperPoint. These are distinctive image features — corners, junctions, texture patches — that can be reliably found in both camera images.

02

Match features across cameras

Match corresponding features between adjacent camera images using descriptor matching (BFMatcher or FLANN). Apply RANSAC to reject outliers. The remaining inlier matches are the same ground points seen from both cameras.

03

Estimate residual misalignment

After applying the nominal geo-registration, matching features should appear at the same ground coordinates in both cameras. Any systematic offset between matched pairs indicates a boresight calibration error for that camera pair. Compute the homography matrix H that best aligns the two sets of matched points.

04

Apply correction and update boresight model

Apply the estimated homography as a warp correction during mosaicking. Slowly update the stored boresight calibration parameters with a moving average to track gradual drift without overreacting to noise.

python
import cv2
import numpy as np

def compute_alignment_correction(img_a, img_b, overlap_mask):
    """
    Estimate homography to align camera B's overlap zone to camera A.
    img_a, img_b  : orthorectified tiles for adjacent cameras
    overlap_mask  : binary mask of the overlap region
    returns       : 3×3 homography matrix H (or None if failed)
    """
    # Crop to overlap zone
    a_crop = cv2.bitwise_and(img_a, img_a, mask=overlap_mask)
    b_crop = cv2.bitwise_and(img_b, img_b, mask=overlap_mask)

    # Convert to grayscale
    a_gray = cv2.cvtColor(a_crop, cv2.COLOR_BGR2GRAY)
    b_gray = cv2.cvtColor(b_crop, cv2.COLOR_BGR2GRAY)

    # Detect and match features
    orb = cv2.ORB_create(nfeatures=500)
    kp_a, des_a = orb.detectAndCompute(a_gray, None)
    kp_b, des_b = orb.detectAndCompute(b_gray, None)

    if des_a is None or des_b is None:
        return None

    matcher = cv2.BFMatcher(cv2.NORM_HAMMING, crossCheck=True)
    matches = matcher.match(des_a, des_b)
    matches = sorted(matches, key=lambda m: m.distance)[:50]

    if len(matches) < 10:
        return None

    pts_a = np.float32([kp_a[m.queryIdx].pt for m in matches])
    pts_b = np.float32([kp_b[m.trainIdx].pt for m in matches])

    # RANSAC homography estimation
    H, mask = cv2.findHomography(pts_b, pts_a, cv2.RANSAC, 5.0)
    inliers = mask.ravel().sum()

    if inliers < 8:
        return None

    return H, inliers

TARGET
TRACKING

Once the mosaic is assembled, motion detection and tracking extract the operationally useful intelligence — the positions and trajectories of all moving objects.

MOTION DETECTION

The simplest and most effective approach for WAMI is frame differencing: subtract the previous frame from the current frame. Moving objects produce non-zero difference values; static background produces near-zero values. Threshold the difference, apply morphological operations to clean up noise, and find connected components — each connected component is a detected mover.

BACKGROUND MODEL

For persistent surveillance, maintain a rolling background model — a time-averaged mosaic of the scene that updates slowly. Subtract the background model from each new frame. This handles slow changes (shadows moving, parked cars) without triggering false detections.

KALMAN FILTER TRACKING

A Kalman filter maintains a probabilistic estimate of each tracked object's position and velocity, updated with each detection. This handles missed detections (object temporarily occluded) and ambiguous associations (two objects close together).

// Kalman state vector for ground target tracking
state = [x, y, vx, vy] // position (m) and velocity (m/s) in world coords Predict: x_pred = F × x_prev P_pred = F × P_prev × Fᵀ + Q // F = motion model, Q = process noise Update: K = P_pred × Hᵀ × (H × P_pred × Hᵀ + R)⁻¹ // Kalman gain x = x_pred + K × (z - H × x_pred) // z = measurement P = (I - K × H) × P_pred

30-DAY ARCHIVE AND FORENSIC REPLAY

The mosaic is continuously written to a rolling circular buffer — 30 days of full-resolution imagery. The database is indexed by time and geographic location, enabling operators to query: "show me what happened at this location between 14:00 and 15:00 yesterday". Track history is stored separately and linked to the video archive for forensic analysis.

GROUND
CALIBRATION

Before first flight, and periodically thereafter, the payload must be calibrated on the ground to determine the intrinsic and extrinsic parameters of each camera.

INTRINSIC CALIBRATION (per camera)

Determines focal length, principal point, and distortion coefficients for each lens/sensor combination. Use a checkerboard target and OpenCV's calibrateCamera() function. Capture 20–30 images of the checkerboard at different positions and angles. This gives you the k1, k2, p1, p2 distortion coefficients you need for undistortion.

python
import cv2
import numpy as np
import glob

# Checkerboard dimensions
BOARD_W, BOARD_H = 9, 6
criteria = (cv2.TERM_CRITERIA_EPS + cv2.TERM_CRITERIA_MAX_ITER, 30, 0.001)

# Prepare object points (0,0,0), (1,0,0), (2,0,0) ...etc
objp = np.zeros((BOARD_W * BOARD_H, 3), np.float32)
objp[:, :2] = np.mgrid[0:BOARD_W, 0:BOARD_H].T.reshape(-1, 2)

objpoints, imgpoints = [], []

for fname in glob.glob('cal_images/cam0/*.png'):
    img = cv2.imread(fname)
    gray = cv2.cvtColor(img, cv2.COLOR_BGR2GRAY)
    ret, corners = cv2.findChessboardCorners(gray, (BOARD_W, BOARD_H), None)
    if ret:
        objpoints.append(objp)
        corners2 = cv2.cornerSubPix(gray, corners, (11,11), (-1,-1), criteria)
        imgpoints.append(corners2)

ret, K, dist, rvecs, tvecs = cv2.calibrateCamera(
    objpoints, imgpoints, gray.shape[::-1], None, None
)
print(f"Camera matrix:\n{K}")
print(f"Distortion: {dist}")
print(f"RMS reprojection error: {ret:.4f} pixels")
# Target: RMS < 0.5 pixels

np.save('cam0_K.npy', K)
np.save('cam0_dist.npy', dist)

EXTRINSIC CALIBRATION (inter-camera)

Determines the 3D rotation and translation between each camera and the IMU reference frame (the boresight). Place the payload on a known flat surface, capture images of a large calibration target with known ground coordinates, and use the projection equations to solve for each camera's orientation relative to the IMU. This must be redone if any camera is dismounted and remounted.

BORESIGHT VERIFICATION FLIGHT

After ground calibration, fly the payload over a scene with easily-identified features at known coordinates (road intersections, building corners from a map). Check that the geo-registered mosaic correctly places features at their known positions. Residual errors indicate boresight calibration error and feed back into the extrinsic calibration refinement.

// Target Accuracy

For a well-calibrated system at 1km AGL: geo-registration accuracy should be <1 pixel (<35cm) for static features. Tracking accuracy for moving targets depends on IMU quality but typically achieves 1–3 pixel RMS (0.35m–1m) across the mosaic.
